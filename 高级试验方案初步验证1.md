# 高级试验方案初步验证 1

## 1. 本次验证目标

本次只验证最容易闭环的一项能力：

- **区间 A 所需的“真实存储 + Xerxes 访问延迟”**

这里不验证：

- Linux guest / x86 FS；
- 用户态 `mmap`；
- 区间 B 的 coherence 维护延迟；
- CXLKV 接入。

本次目标是尽可能沿用 Xerxes 作者已经提供的最小 example，简单确认下面这条路径确实成立：

```text
请求 -> XerxesWrapper -> Xerxes 拓扑计算额外延迟 -> gem5 MemCtrl 真实读写
```

如果这个验证成功，就至少说明：

1. XerxesWrapper 已经能作为 gem5 内存控制器前的一层延迟包装器工作。
2. 请求不是只“算了个假延迟”，而是真的落到 gem5 模拟内存里。
3. 区间 A 的核心 building block 是存在的。

---

## 2. 对应的现成代码

本次验证直接使用作者给的最小示例：

- `CXL_BETTER_SIM/Xerxes/gem5/configs/example/xerxes_wrapper_minimal.py`

这个脚本的特点：

- `Root(full_system=False, system=system)`，所以它是 `SE` 风格的最小 smoke test，不是 FS。
- `system.mem_ctrl = XerxesWrapper()`。
- `system.mem_ctrl.use_snoop = False`。
- `system.mem_ctrl.dram = DDR3_1600_8x8(range=system.mem_ranges[0])`。
- 前端流量来自 `MemTest`，不是 Linux 程序。

这正好对应我们这次要验证的区间 A：

```text
真实存储 + Xerxes 访问延迟
不带 coherence 维护
```

---

## 3. 验证思路

我们不去证明“延迟值一定精确等于某个理论公式”，而是先证明下面三件事：

1. **能跑通**
   - gem5 能成功实例化 `XerxesWrapper`。
   - 示例能正常退出，不是初始化阶段就报错。

2. **真的经过 Xerxes**
   - 输出目录里会生成 Xerxes 的请求日志，例如 `xerxes_log.csv`。
   - 日志里能看到请求经过 Xerxes 设备并产生非零延迟。

3. **后端仍然是真实内存**
   - `MemTest` 能正常完成若干 load/store，而不是因为没有真实 backing memory 而失败。

只要这三件事成立，就足以支持“区间 A 的最小原型可行”这个判断。

---

## 4. 推荐执行步骤

## 4.1 确认子模块就绪

在仓库根目录执行：

```bash
cd /root/code/cxlkv
git submodule update --init --recursive CXL_BETTER_SIM/Xerxes/third_party/gem5
```

如果你是第一次在这台机器上使用 Xerxes，建议也把 Xerxes 自己依赖的子模块补齐：

```bash
git submodule update --init --recursive CXL_BETTER_SIM/Xerxes
```

说明：

- `apply_gem5_overlay.sh` 依赖 `CXL_BETTER_SIM/Xerxes/third_party/gem5` 存在。
- Xerxes build 过程中会用到 `ext/xerxes` 指向 Xerxes 仓库本体。

---

## 4.2 应用 Xerxes 对 gem5 的 overlay

执行：

```bash
cd /root/code/cxlkv/CXL_BETTER_SIM/Xerxes
./scripts/apply_gem5_overlay.sh
```

预期输出应类似：

```text
Applied Xerxes gem5 overlay to .../third_party/gem5
Linked .../third_party/gem5/ext/xerxes -> ../../..
```

这一步的作用：

- 把 `gem5/` 目录下 Xerxes 维护的 overlay 文件复制到 `third_party/gem5`；
- 建立 `third_party/gem5/ext/xerxes` 符号链接；
- 让后续 gem5 构建能看到 `XerxesWrapper`、`CohInterface` 以及协议扩展。

---

## 4.3 构建用于最小验证的 gem5 二进制

最小示例 `xerxes_wrapper_minimal.py` 使用的是 `X86_MI_example_Xerxes` 构建目标。

推荐直接执行 Xerxes 自带脚本：

```bash
cd /root/code/cxlkv/CXL_BETTER_SIM/Xerxes
./scripts/build_gem5_xerxes_opt.sh X86_MI_example_Xerxes
```

如果你想直接在 gem5 目录手工构建，对应命令是：

```bash
cd /root/code/cxlkv/CXL_BETTER_SIM/Xerxes/third_party/gem5
scons defconfig build/X86_MI_example_Xerxes build_opts/X86_MI_example_Xerxes
scons -j"$(nproc)" build/X86_MI_example_Xerxes/gem5.opt --ignore-style
```

成功后，目标二进制应位于：

- `CXL_BETTER_SIM/Xerxes/third_party/gem5/build/X86_MI_example_Xerxes/gem5.opt`

---

## 4.4 运行作者提供的最小 example

执行：

```bash
cd /root/code/cxlkv/CXL_BETTER_SIM/Xerxes/third_party/gem5
build/X86_MI_example_Xerxes/gem5.opt \
  -d m5out/xerxes_no_coh_traffic \
  configs/example/xerxes_wrapper_minimal.py \
  --max-loads=8 \
  --progress-interval=4
```

说明：

- `--max-loads=8` 让测试很快结束，适合 smoke test。
- `-d m5out/xerxes_no_coh_traffic` 指定输出目录。
- 这个 example 本身就是 Xerxes 文档中建议的 wrapper-only smoke test。

## 5. 如何判断验证成功

## 5.1 最基本的成功标准

仿真进程正常退出，没有在初始化或运行阶段报错。

终端最后应出现类似：

```text
Exiting @ tick ... because ...
```

或最小脚本中的退出原因打印。

如果一开始就出现下面这类报错，则说明环境还没搭好：

- 找不到 `XerxesWrapper` SimObject；
- 找不到 `ext/xerxes`；
- build target 不匹配；
- overlay 没应用成功。

## 5.2 输出目录里应出现 Xerxes 日志

检查输出目录：

```bash
cd /root/code/cxlkv/CXL_BETTER_SIM/Xerxes/third_party/gem5
find m5out/xerxes_no_coh_traffic -maxdepth 2 -type f | sort
```

重点关注：

- `m5out/xerxes_no_coh_traffic/xerxes_log.csv`
  - 或脚本/版本差异下位于输出目录内部的同名文件。

如果没有这个日志，说明请求大概率没有真正通过 XerxesWrapper 的日志路径。

## 5.3 查看 Xerxes 请求日志

执行：

```bash
cd /root/code/cxlkv/CXL_BETTER_SIM/Xerxes/third_party/gem5
sed -n '1,40p' m5out/xerxes_no_coh_traffic/xerxes_log.csv
```

预期会看到 CSV 表头和若干请求记录，字段大致类似：

```text
srcname,dstname,type,addr,sent,arrive,lat,payload
...
```

对这份日志，至少应确认：

1. 有多条请求记录，不是空文件。
2. `lat` 字段非零。
3. 请求确实被 Xerxes 记录了，而不是完全绕开 Xerxes。

## 5.4 查看 gem5 配置中是否启用了 `use_snoop = False`

这一步不是必须，但建议确认一下，以免误跑到 B 路径。

可以直接看脚本：

```bash
sed -n '1,120p' /root/code/cxlkv/CXL_BETTER_SIM/Xerxes/gem5/configs/example/xerxes_wrapper_minimal.py
```

关键点：

- `system.mem_ctrl = XerxesWrapper()`
- `system.mem_ctrl.use_snoop = False`

这说明此次验证的确只覆盖 A 区间能力，而没有引入 coherence path。

## 6. 建议额外做的一个小对照

如果想让“Xerxes 确实在加延迟”这件事更直观，建议做一个小对照实验：

1. 保持 `xerxes_wrapper_minimal.py` 不变，跑一遍默认参数。
2. 在脚本里把 `system.mem_ctrl.cxlLatency` 调得更大，例如从默认值提高若干倍，再跑一遍。
3. 比较：
   - 退出 tick；
   - `xerxes_log.csv` 里的 `lat`；
   - 如果有 stats，再比较平均访问时间。

这不是严格定量验证，但能帮助确认：

- 请求额外延迟确实受 XerxesWrapper 参数影响；
- 不是单纯 DRAM latency 在起作用。

注意：

- 如果要改脚本做这个对照，请改副本，不要污染作者原始 example。

---

## 7. 本次验证通过后，可以得到的结论

如果上面的最小验证成功，可以合理得出以下结论：

1. Xerxes 已经可以包装在 gem5 内存路径前端，对普通访存插入额外延迟。
2. XerxesWrapper 后端仍然连接 gem5 的真实模拟内存，因此“真实存储”这一点成立。
3. 区间 A 所需的最核心 building block 已具备。

但此时**还不能**推出下面这些更强结论：

1. 还不能说明 `x86 FS + Linux guest + mmap` 已经可用。
2. 还不能说明双区间系统已可用。
3. 还不能说明区间 B 的 coherence 维护路径成熟可用。
4. 还不能说明可以立刻接入 CXLKV。

所以这一步的定位应该是：

- **验证区间 A 的最小可行性**
- 不是验证完整高级方案

---

## 8. 如果失败，优先检查什么

建议按下面顺序排查：

1. `third_party/gem5` 子模块是否存在。
2. `./scripts/apply_gem5_overlay.sh` 是否执行成功。
3. `third_party/gem5/ext/xerxes` 是否已正确链接到 Xerxes 根目录。
4. 是否真的构建了 `X86_MI_example_Xerxes`，而不是普通 gem5 target。
5. 是否使用了 `configs/example/xerxes_wrapper_minimal.py`，而不是别的脚本。
6. 输出目录里是否有 `xerxes_log.csv`。

---

## 9. 建议的下一步

如果“初步验证 1”通过，下一步建议不是马上进 FS，而是先做一个更贴近目标的中间验证：

- 在 `SE` 下做一个**双区间 smoke test**
  - A：`use_snoop=false`
  - B：`use_snoop=true`

这样可以把风险拆成两层：

1. 先确认 A/B 双路径在 synthetic 流量下都能工作；
2. 再进入更重的 `x86 FS + guest mmap` 工程。

这会比一上来直接做 Linux guest 双池系统更稳。
