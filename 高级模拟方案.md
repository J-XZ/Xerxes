# 高级模拟方案（FS 双可缓存内存池）

本文只讨论并落实一个目标方案：
- 使用 gem5 **Full-System (FS)** 运行 Linux；
- 多进程/多核分别代表多个计算节点角色；
- 使用两个独立、都可被 CPU 缓存的内存池：
  - `POOL_NC`：可缓存，但不模拟 CPU 间一致性维护的额外开销；
  - `POOL_C`：可缓存，并额外模拟 CPU 间一致性维护开销。

> 关键修正：这里的 `NC` 不表示 Linux 映射为 uncached，也不表示 CPU 不能缓存；它表示实验模型中“不引入跨 CPU 一致性维护额外成本”。两个池都应保持 cacheable，差异主要来自 `XerxesWrapper`/`Snoop`/`CohInterface` 是否注入一致性相关开销。

---

## 1. 最终结论

在当前 Xerxes + gem5 集成代码基础上，该 FS 方案 **有条件可实现**。

必须明确分工：
- gem5 FS 负责真实 Linux、页表、cache、load/store 和内存数据保存；
- gem5 `MemCtrl`/backing store 负责真实数据读写；
- Xerxes/DRAMsim3 standalone 本身不保存真实数据，只适合做 timing 事务模拟；
- `XerxesWrapper` 在 gem5 内存访问路径中插入 CXL/bus/snoop/coherence timing 开销，然后回到 gem5 内存控制器完成真实读写。

因此，本方案不能只依赖 standalone 路径：

```text
Requester -> Switch -> Bus -> DRAMsim3Interface
```

而应依赖 FS 路径：

```text
Linux 用户程序
  -> mmap 两个物理窗口
  -> gem5 CPU/cache/MMU 执行真实 load/store
  -> XerxesWrapper 注入 CXL/bus/snoop/coherence 开销
  -> gem5 MemCtrl/backing store 保存真实数据
```

主要工作量集中在：
1. FS 启动脚本与系统拓扑搭建；
2. 双内存池物理地址窗口与 `XerxesWrapper` 路由；
3. guest kernel driver 暴露两个 cacheable mmap 区域；
4. 多进程测试程序与实验自动化。

整体属于“中等偏大”工程，但路径清晰，可分阶段落地。

---

## 2. 为什么选择“两个独立内存池”

相对“同一内存池分两段”，两个独立池更容易的点：

1. 配置与路由更直观
- 直接创建两个 `XerxesWrapper`，每个绑定独立地址范围和参数。
- 一个 wrapper 开启一致性开销路径，另一个关闭，逻辑上天然隔离。

2. 降低协议耦合
- coherent/non-coherent 行为不在同一个 wrapper 内分支处理，减少条件路径和隐藏 bug。

3. 结果更容易解释
- 日志和统计可以按池独立归因，不容易混淆。

4. 迭代更稳
- 可以先让 `POOL_NC` 跑通真实读写，再单独压测 `POOL_C` 的一致性开销。

因此本文固定采用：**双池方案**。

## 3. 目标架构（FS）

## 3.1 计算节点抽象

- 一个 gem5 FS `System`；
- 多核 CPU（例如 `N` 个 core）；
- Linux 启动后运行 `N` 个进程；
- 每个进程可视为一个“独立计算节点角色”（逻辑独立，非独立 OS 实例）。

边界说明：
- 这是单 Linux kernel 下的多核/多进程模型；
- 它能评估共享池访问、cacheable 映射、同步开销和一致性开销；
- 它不是多个独立 OS 实例，也不直接等价于真实多主机 CXL fabric。

## 3.2 内存池抽象

两个池都应是 **CPU cacheable**：

- `POOL_NC`：`XerxesWrapper(use_snoop=False)`，cacheable 映射，真实数据由 gem5 内存保存，不额外注入 CPU 间一致性维护成本；
- `POOL_C`：`XerxesWrapper(use_snoop=True, coh_interface=...)`，cacheable 映射，在访问路径中注入 snoop/back-invalidation/coherence 相关成本。

每个池有独立物理地址窗口：
- `POOL_NC_BASE`, `POOL_NC_SIZE`
- `POOL_C_BASE`, `POOL_C_SIZE`

推荐地址布局示例：

```text
0x0000_0000 - 0x7fff_ffff    普通 Linux RAM，例如 2GB
0x8000_0000 - 0xbfff_ffff    POOL_NC，例如 1GB，cacheable
0xc000_0000 - 0xffff_ffff    POOL_C，例如 1GB，cacheable + coherence overhead
```

gem5 中挂三个地址范围：

```python
system.mem_ranges = [
    AddrRange("0x00000000", size="2GiB"),
    AddrRange("0x80000000", size="1GiB"),
    AddrRange("0xc0000000", size="1GiB"),
]
```

`SystemXBar` 按地址范围把请求转发到对应内存控制器：

```text
normal RAM -> 普通 MemCtrl
POOL_NC    -> mem_ctrl_nc: XerxesWrapper(use_snoop=False)
POOL_C     -> mem_ctrl_c:  XerxesWrapper(use_snoop=True, coh_interface=...)
```

## 3.3 用户态访问方式

用户态不能凭空 `mmap` 任意物理地址。推荐实现一个很小的 guest kernel driver：

- `/dev/cxl_pool_nc`：映射 `POOL_NC`；
- `/dev/cxl_pool_c`：映射 `POOL_C`；
- `mmap()` 中用固定 PFN 把池映射到用户态；
- 两个池都使用 cacheable page protection；
- `ioctl()` 可返回 base/size、清零、统计和实验参数。

用户程序接口：

```c
int fd_nc = open("/dev/cxl_pool_nc", O_RDWR);
void *pool_nc = mmap(NULL, nc_size, PROT_READ | PROT_WRITE, MAP_SHARED, fd_nc, 0);

int fd_c = open("/dev/cxl_pool_c", O_RDWR);
void *pool_c = mmap(NULL, c_size, PROT_READ | PROT_WRITE, MAP_SHARED, fd_c, 0);
```

启动 Linux 时需要避免这两个物理窗口被普通内核页分配器占用。可用内核参数将窗口保留：

```text
memmap=1G$0x80000000 memmap=1G$0xc0000000
```

注意：具体命令行里 `$` 可能需要转义，取决于 gem5 启动脚本和 shell 传参方式。

## 4. 可行性分析（按需求逐条）

## 4.1 真实数据存储

可行性：中高。

原因：
- standalone Xerxes/DRAMsim3Interface 不保存真实数据；
- gem5 FS 路线中，真实 load/store 由 gem5 CPU/cache/MMU/MemCtrl 处理；
- `XerxesWrapper` 只改变访问时序和一致性相关路径，最后仍回到 gem5 `MemCtrl` 完成真实读写。

结论：
- 不能只用 standalone Xerxes；
- 必须走 gem5 FS + `XerxesWrapper` + gem5 memory backing store。

## 4.2 FS + Linux + mmap 语义真实度

可行性：中高。

原因：
- gem5 FS 提供真实内核、页表、系统调用路径；
- `mmap`/缺页/页权限等行为由 Linux 路径处理；
- 需要 guest driver 显式暴露两个物理池，不能依赖用户态直接映射物理地址。

结论：满足“真实 mmap 语义”的目标，但需要实现 driver。

## 4.3 多进程/多核表示多计算节点

可行性：高。

原因：
- FS 下可启动多核并运行多个用户进程；
- 可用 `taskset`/cpuset 绑定核，控制节点角色与并发。

边界说明：
- 这是“逻辑独立节点”，不是“多个独立 OS 实例”；
- 对共享池访问、cacheable 映射与一致性开销评估通常已经足够。

## 4.4 两个池都可缓存，但一致性开销不同

可行性：中。

原因：
- `POOL_NC` 和 `POOL_C` 的用户态映射都保持 cacheable；
- `POOL_NC` 走 `XerxesWrapper(use_snoop=False)`，作为低开销基线；
- `POOL_C` 走 `XerxesWrapper(use_snoop=True, coh_interface=...)`，注入一致性维护相关开销。

边界说明：
- 当前模型更适合表达“同样可缓存的数据访问，C 池额外付出一致性维护成本”；
- 如果未来要精确模拟“多个独立主机各自缓存同一地址、但硬件不维护一致性导致读到陈旧值”，需要更深的 gem5 cache/coherence protocol 改造，不属于第一阶段。

## 4.5 运行自定义可执行文件

可行性：高。

原因：
- FS 下可直接运行你的 ELF；
- 程序通过 driver `mmap` 将两池映射到进程地址空间；
- 程序内可以实现真实读写、同步、工作负载分布和数据正确性校验。

## 5. 详细执行计划（FS only）

## Phase 0：实验契约与地址布局（先定规范）

目标：避免后续接口反复变更。

1. 固化参数
- `NUM_NODES`（进程/核心数）
- `POOL_NC_SIZE`, `POOL_C_SIZE`
- `POOL_NC_BASE`, `POOL_C_BASE`
- cache line 与页大小约束
- 两池均为 cacheable 映射

2. 固化地址布局
- 普通 Linux RAM 独立于两个 pool；
- `POOL_NC` 和 `POOL_C` 用 `memmap=` 或等价方式保留；
- guest driver 只映射保留窗口，不参与普通页面分配。

3. 固化程序接口
- 通过命令行/环境变量传入两池长度、节点 ID、访问模式；
- driver `ioctl()` 返回池 base/size/version；
- 统一统计输出格式（每节点吞吐、平均延迟、p95/p99、校验错误数）。

4. 固化实验指标
- 池级：带宽、平均延迟、尾延迟；
- 进程级：吞吐、延迟、失败重试；
- 正确性：读回校验、跨进程可见性检查、同步变量一致性。

交付：`实验参数表 + 地址布局图 + 用户态/driver ABI`。

## Phase 1：FS 最小可启动基线

目标：先跑通 FS，不接双池复杂逻辑。

1. 准备 FS 内核与 rootfs（与当前 gem5 版本匹配）。
2. 新建配置脚本：`gem5/configs/example/xerxes_fs_dualpool.py`。
3. 验证 Linux 正常启动，能执行一个简单用户程序。
4. 确认串口、init 脚本、文件系统挂载和结果导出路径稳定。

验收：
- 进入用户态；
- 程序可执行；
- 基础串口日志稳定；
- gem5 stats 可正常生成。

## Phase 2：接入双可缓存内存池（核心）

目标：在同一 FS 系统中挂两个 Xerxes 池。

1. 创建普通 RAM 控制器
- 用于 Linux kernel/rootfs/普通 heap/stack；
- 不参与 pool 实验统计。

2. 创建两个 `XerxesWrapper`
- `mem_ctrl_nc`：`use_snoop=False`，较低 `cxlLatency` 或仅保留 bus/timing；
- `mem_ctrl_c`：`use_snoop=True`，接 `CohInterface`，体现一致性维护额外成本。

3. 分配互不重叠地址窗口
- `AddrRange(POOL_NC_BASE, size=POOL_NC_SIZE)`
- `AddrRange(POOL_C_BASE, size=POOL_C_SIZE)`

4. 路由接线
- 确认 XBar/内存路由按地址范围分发到正确 pool；
- 为两个 wrapper 配置不同 `log_name`，便于归因。

5. Ruby/CohInterface 连接
- `POOL_C` 绑定 `CohInterface.ruby_port` 到 Ruby path；
- `POOL_NC` 不绑定一致性维护路径。

验收：
- 访问 `POOL_NC` 地址只进入 `mem_ctrl_nc` 日志；
- 访问 `POOL_C` 地址只进入 `mem_ctrl_c` 日志；
- 两池均可 cacheable 访问；
- `POOL_C` 可观察到额外一致性相关统计或日志差异。

## Phase 3：guest kernel driver mmap 接入

目标：让用户程序在 FS 内直接访问双池。

1. 实现 guest driver：`cxl_pool.ko`
- 两个字符设备：`/dev/cxl_pool_nc`、`/dev/cxl_pool_c`；
- `mmap()` 将固定物理窗口映射到用户态；
- 两个映射都保持 cacheable；
- `ioctl()` 支持查询池信息、清零池、简单健康检查。

2. Linux 启动参数
- 使用 `memmap=` 或等价方式保留两段物理窗口；
- 在启动日志和 `/proc/iomem` 中确认窗口未被普通内存分配器占用。

3. 最小测试程序
- 分别 mmap 两个池；
- 写入 magic pattern；
- 读回校验；
- fork 多进程后检查跨进程可见性。

验收：
- 两池均能真实保存数据；
- 进程退出重启后，在同一次仿真运行内可读回预期内容；
- 多进程共享映射可见。

## Phase 4：多进程并发与工作负载

目标：把“池能访问”升级为“能跑实验”。

1. 多进程并发
- 启动 `NUM_NODES` 个进程；
- 每进程绑定核心并设置节点角色；
- 每个进程同时 mmap `POOL_NC` 和 `POOL_C`。

2. 访问模式
- 顺序读写；
- 随机读写；
- 热区读写；
- 共享计数器/锁；
- 树索引或 CXLKV 子集工作负载。

3. 数据采集
- 用户程序日志；
- gem5 stats；
- 两个 Xerxes wrapper 独立日志；
- Ruby/CohInterface 统计。

验收：
- 程序稳定运行；
- 双池性能曲线可区分；
- `POOL_C` 在并发共享访问下呈现额外一致性维护开销；
- 正确性校验无误。

## Phase 5：实验自动化与复现

1. 自动化脚本
- 一键 sweep：`NUM_NODES / POOL_NC_SIZE / POOL_C_SIZE / 访问模式 / 读写比`。

2. 报告脚本
- 自动生成对比图：
  - NC vs C 平均延迟；
  - NC vs C p95/p99；
  - 节点数扩展曲线；
  - coherence overhead 占比。

3. 固化回归
- 功能回归：mmap、读写校验、多进程共享；
- 性能趋势回归：C 池开销应随共享写/冲突增强而上升。

验收：
- 给定参数可重复得到同趋势结果；
- 输出目录结构稳定，便于论文/报告复现。

## 6. 实施优先级（建议顺序）

1. Phase 1（FS 启动）
2. Phase 2（双池接线）
3. Phase 3（guest driver + cacheable mmap）
4. Phase 4（多进程工作负载）
5. Phase 5（自动化）

说明：先确保“真实存数据、能 mmap、能路由、能测”，再逐步增强一致性开销模型。

## 7. 风险与缓解

1. `mmap` 物理窗口实现风险
- 风险：用户态不能直接映射任意物理地址，保留窗口和 page protection 配置不当会导致崩溃或被 Linux 普通内存分配器占用。
- 缓解：实现 guest driver；启动参数保留 pool；用 `/proc/iomem`、driver probe 日志和读写校验确认。

2. cacheable 映射属性不符合预期
- 风险：driver 使用了 uncached/device mapping，导致两个池不再都是 CPU cacheable。
- 缓解：明确禁止 `pgprot_noncached()`；保留默认 cacheable page protection；在文档和代码注释中固定该语义。

3. coherent 池并发瓶颈
- 风险：在极高并发下一致性路径可能成为吞吐瓶颈，甚至暴露 `CohInterface` 当前阶段的并发限制。
- 缓解：先控制并发规模；从 1/2/4 核逐步扩展；必要时增强 `CohInterface` outstanding 能力。

4. 指标归因混淆
- 风险：程序、OS、cache、互连、DRAM、一致性开销混在一起。
- 缓解：普通 RAM 与两个 pool 分离；两个 wrapper 独立日志；统一时间戳；按层拆分指标。

5. `POOL_NC` 语义被误解
- 风险：把 `NC` 理解为 uncached，导致实验实现偏离目标。
- 缓解：本文固定定义 `POOL_NC = cacheable but no extra inter-CPU coherence overhead`；命名保留是为了对应实验概念，但实现中必须注明两个池都 cacheable。

## 8. 未来可选增强（仅建议）

1. 更真实的多主机非一致性模型
- 若未来需要模拟“不同主机缓存同一地址但硬件不维护一致性，因此可能读到陈旧值”，需要改 gem5 cache/coherence protocol 或建立多系统实例模型。
- 这比当前双池 timing-overhead 模型复杂很多，不属于第一阶段。

2. 增强 `CohInterface`
- 支持更多 outstanding invalidation；
- 增加 per-line/per-core 统计；
- 更细粒度地区分 invalidate、writeback、ownership transfer 开销。

3. CXLKV 专用 workload
- 将当前树索引/CXLKV 的读写路径抽象成 FS 用户态 workload；
- 对 `POOL_NC`/`POOL_C` 分别部署不同数据结构区域；
- 输出结构级指标，例如 merge 次数、delta 命中率、尾延迟。

关于现有的多核模拟的问题：

不是单纯“性能不行”。现有代码做受控多核原型可以，但真实 FS 多核压力下可能 fatal/卡住/协议不完整。

主要硬限制：

CohInterface 只支持一个 outstanding invalidation；并发触发会 fatal_if(pendingPkt || outstanding)。
XerxesWrapper 只支持 INV burst == 1，否则 fatal。
use_snoop=True 但 coh_interface 没接好，第一次 INV 会 fatal。
现有 README 也把它称为示例/phase-1 路径，不是完整高并发 FS 产品级实现。
所以：不是“一开多核就必崩”，而是多核并发下很容易触发当前 phase-1 限制。 小规模、受控 workload 可以验证；高并发真实 workload 需要加固。
