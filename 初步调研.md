## 2. 整体架构（基于源码）

### 2.1 核心对象与职责
- `Simulation`：全局仿真容器，持有 `Topology` 和 `System`。
- `Topology`：拓扑图（节点、边、路由表）。路由默认通过 BFS 构建 next-hop。
- `System`：设备注册表，维护 `TopoID -> Device*`。
- `Device`：所有设备基类（Requester、Switch、DuplexBus、Snoop、DRAMsim3Interface）。
- `EventEngine`（在 `xerxes_standalone.cc` 中）：全局事件队列（按 Tick 的 multimap）。

### 2.2 运行主流程
入口 `main.cc` 的主流程：
1. 创建 `Simulation`，并调用 `init_sim()` 设置全局仿真对象。
2. 解析 TOML 配置（`parse_config()`）。
3. 为所有 Requester 注册初始 issue 事件。
4. 循环调用 `step()` 推进事件引擎；并按 `clock_granu` 推进 DRAMsim3 时钟。
5. 所有请求发完、队列排空后退出，输出统计。

### 2.3 配置解析与设备装配
`parse_config()` 会：
- 从 TOML 读取 `general.devices` 与 `general.edges`。
- 动态实例化设备（支持 `Requester/Switch/DuplexBus/DRAMsim3Interface/Snoop`）。
- 把设备加入 `System`，记录名字到 `TopoID` 映射。
- 根据 `edges` 建图，并调用 `topology->build_route()`。
- 把所有 DRAM 端点信息（起始地址、容量、写比例）注入每个 Requester 的 end points。

### 2.4 关键设备行为
- `Requester`：支持 `stream/random/hotcold/trace` 负载生成；内部有 issue queue 与 fake LRU cache；请求发出后经拓扑转发。
- `Switch`：多端口转发，端口内有按上游来源分队列与时间线调度，统计 switch queue delay/time。
- `DuplexBus`：支持全双工/半双工，建模方向切换开销（half duplex reverse）、帧封装和带宽延迟。
- `Snoop`：挂在内存前用于一致性相关行为，含可替换驱逐策略（LRU/FIFO/LFI 等）。
- `DRAMsim3Interface`：对接 DRAMsim3，负责请求接入、可受理检查、回调完成并发响应包。

### 2.5 数据与日志
- 包模型：`Packet`（请求类型、时戳、路径、统计字段）。
- 统计字段：bus/switch/snoop/DRAM 细分延迟，最终可汇总 `total_time`。
- 默认日志：CSV，每行一个请求，字段在 README 有完整解释。

### 8.0 先跑哪个作者示例，以及它在做什么

建议先跑两个官方示例：`sample-bus` 和 `sample-topo`。原因是它们分别覆盖了 Xerxes 最核心的两类实验维度，最适合“先跑通，再调试”。

1. `sample-bus`（总线参数示例）
   - 生成脚本：`configs/sample-bus.py`
   - 主要目的：在拓扑基本不变时，调整总线相关参数（如 frame size、读写比例）并观察带宽/时延变化。
   - 你能验证的关键点：
     - 请求链路是否完整走通（Requester -> Bus/Switch/Snoop -> DRAM -> Response）
     - `bus_queuing`、`bus_time`、`total_time` 等字段是否随参数变化而变化

2. `sample-topo`（拓扑对比示例）
   - 生成脚本：`configs/sample-topo.py`
   - 主要目的：在负载模型基本一致时，切换拓扑（如 ring/chain/full）并比较路由与排队行为差异。
   - 你能验证的关键点：
     - `parse_config()` 和 `topology->build_route()` 是否正确构图与建路由
     - `switch_queuing`、`switch_time`、`total_time` 是否体现不同拓扑特征

为什么先推荐这两个：
- 它们是 README 的官方入口，复现实验路径最短。
- 配置生成和结果输出都很直观，适合快速确认环境与模型是否正常。
- 很适合配合 gdb/lldb 打断点，覆盖“配置加载 -> 事件调度 -> 设备转发 -> DRAM 回调 -> 统计输出”的完整链路。

### 8.3 生成作者示例配置并先跑一遍

先解释你当前这条命令在做什么（以 `sample-bus.py` 为例）：

```bash
python3 configs/sample-bus.py \
  --scale 4 \
  --ratio 0.25 \
  --log output/sample-bus.csv \
  > configs/sample-bus.toml
```

这条命令的含义：
1. 运行配置生成脚本 `configs/sample-bus.py`。
2. `--scale 4`：生成 4 对 Host/Mem（即 4 个 Requester + 4 个 DRAMsim3Interface）。
3. `--ratio 0.25`：把每个内存端点的 `wr_ratio` 设为 0.25（写比例 25%）。
4. `--log output/sample-bus.csv`：把仿真运行时的请求级日志文件名写进配置里。
5. `> configs/sample-bus.toml`：把脚本打印到标准输出的 TOML 配置重定向保存为文件。

输出文件的意义：
1. `configs/sample-bus.toml`：仿真“输入配置”。
  - 定义了 `max_clock`、`log_name`、设备列表、设备参数和拓扑连接边（`edges`）。
  - `build/Xerxes` 或 `debug/Xerxes` 读取它来决定如何搭建和运行模拟系统。
2. `output/sample-bus.csv`：仿真“输出结果”。
  - 每行对应一条请求，记录端到端时延及各组件分解时延（如 `bus_queuing`、`switch_time`、`dram_time`、`total_time`）。
  - 后续 `output/report.py` 就是基于这个 CSV 做统计分析。

脚本内部的基本流程（`sample-bus.py`）：
1. 解析命令行参数（`scale`、`ratio`、`log`）。
2. 按 `scale` 创建 Requester 和 DRAMsim3Interface 设备对象。
这里python脚本只是在配置文件中逻辑创建了这些设备，真正的 C++ 设备对象是在 `Xerxes` 运行时根据 TOML 配置动态实例化的。
3. 创建两个 Switch 和一个 DuplexBus。
这里同样先是 Python 配置对象，最终在 C++ 运行时变成仿真器内的逻辑设备对象：
  - `Switch`：仿真里的多端口转发/汇聚设备，不代表真实以太网交换机，而是 Xerxes 拓扑图上的一个路由节点。它根据 `Topology::next_node()` 选择下一跳端口；每个输出端口内部按上游邻居维护队列，并用 `Timeline::transfer_time()` 计算端口被占用时产生的排队时间。统计上主要贡献 `switch_queuing` 和 `switch_time`。`sample-bus.py` 里把 `Switch-1`、`Switch-2` 的 `delay` 设为 0，所以它们更像“理想汇聚/分发节点”，主要负责把多 Host/多 Mem 接到中间 Bus 两侧。
  - `DuplexBus`：仿真里的共享总线/链路设备，重点建模链路传输开销而不是路由选择。它按 `payload`、`frame_size`、`width`、`delay_per_T` 计算传输时间，并用 `Timeline` 表示链路占用，所以多个请求同时经过时会产生 `bus_queuing`。如果配置成半双工，还会用 `half_rev_time` 建模方向反转开销；默认 `sample-bus.py` 使用全双工 `is_full=true`。
  - 多设备交互：`sample-bus` 的拓扑是 `Host-* -> Switch-1 -> Bus -> Switch-2 -> Mem-*`，响应包再按反向路径回来。`Switch-1` 负责把多个 Host 的请求汇聚到 Bus；`Bus` 负责模拟中间共享链路的带宽/帧化/排队；`Switch-2` 负责把经过 Bus 的请求分发到目标内存端点。反向响应时角色反过来，`Switch-2` 汇聚内存响应，`Bus` 传回，`Switch-1` 分发给对应 Host。
4. 设置关键参数：
  - Requester：`interleave_type=random`、`interleave_param=500`
  - DRAMsim3Interface：`wr_ratio=ratio`
  - Switch：`delay=0`
  - DuplexBus：`frame_size=64`（弱化 framing 开销，突出总线效应）
5. 用 `mkcfg.utils.Config()` 聚合设备并建立连接：
  - Host-* -> Switch-1
  - Switch-1 -> Bus -> Switch-2
  - Switch-2 -> Mem-*
6. `print(cfg)` 输出 TOML 文本（被 shell 重定向写入 `configs/sample-bus.toml`）。

### 8.4 用 gdb 走完整仿真流程

#### 8.4.1 启动 gdb

#### 8.4.2 推荐断点（按流程）
在 gdb 中依次输入：
```gdb
set pagination off
set print pretty on

# 阶段A: 启动与配置解析
break main
break xerxes::parse_config

# 阶段B: 事件引擎推进
break xerxes::step
break xerxes::Device::sched_transit

# 阶段C: 关键设备 transit
break xerxes::Requester::transit
break xerxes::DuplexBus::transit
break xerxes::Snoop::transit
break xerxes::DRAMsim3Interface::transit
break xerxes::DRAMsim3Interface::callback

# 运行
run
```

#### 8.4.3 每阶段建议观察点
阶段 A（main/parse_config）：
```gdb
next
print config_file
continue
```
目标：确认 TOML 文件路径正确，设备/边加载成功。

阶段 B（step/sched_transit）：
```gdb
bt
frame 0
continue
```
目标：确认事件是按 Tick 触发，调度链条正常。

阶段 C（各设备 transit/callback）：
```gdb
info args
info locals
print pkt.id
print pkt.arrive
continue
```
目标：观察请求在 Requester -> Bus/Switch/Snoop -> DRAM -> 响应回程中的关键字段变化。

阶段 D（收尾）：
当回到 `main` 并接近退出前，可检查：
```gdb
continue
```
终端应打印 `Simulation finished.` 并输出统计。

### 8.8 常见问题与排查

1. 子模块没拉全，构建时报 DRAMsim3 相关错误。
  - 处理：`git submodule update --init --recursive`

2. 调试器断点“未解析/不命中”。
  - 原因：用的是 Release 二进制或符号被优化。
  - 处理：执行 `./scripts/build_xerxes_debug.sh` 重新编译。

3. 运行后没有 CSV。
  - 原因：TOML 里 `--log` 路径不可写或目录不存在。
  - 处理：确保 `output/` 存在，并重新生成 TOML。

4. lldb 能启动但变量不可读。
  - 原因：编译优化太高或调试信息不完整。
  - 处理：Debug 构建，必要时加 `-O0 -g3`。
