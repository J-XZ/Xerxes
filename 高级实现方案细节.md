# Xerxes 现状评估与改造建议

## 1. 结论先行

结论分成两层：

1. **如果只问“Xerxes 现在有没有实现所需机制的 building blocks”**，答案是有一部分已经具备。
   - 区间 A 所需的“真实存储 + Xerxes 访问延迟”已经有直接基础。
   - 区间 B 所需的“访问延迟 + back-invalidation / coherence 维护延迟”已经有原型级路径。

2. **如果问“在不修改现有代码的前提下，能不能直接实现需求文档里的完整目标系统”**，答案是**不能**。
   - 现有代码没有现成的 `x86 FS + Linux guest + 用户态 mmap 双区间内存池` 配置入口。
   - 现有 B 区间示例并不是“Linux 进程直接访问该区间”，而是“XerxesWrapper 作为外部 agent 访问一段 alias 地址，再通过 `CohInterface -> Ruby DMA` 去触发对 Ruby 体系内 cache line 的失效”。
   - 现有 coherence 路径明确还是 `phase-1 scaffolding`，并不适合直接拿来当多进程、多 cache line、高并发的完整 HWCC 模型。

因此，**现有代码可以支撑原型验证，但不能零修改落地完整方案**。

---

## 2. 现有代码已经具备的能力

## 2.1 区间 A：真实存储 + Xerxes 访问延迟

这一部分支撑最完整。

直接证据：

- `gem5/src/mem/XerxesWrapper.py`
  - `XerxesWrapper` 直接继承 `MemCtrl`。
- `gem5/src/mem/xerxes_wrapper.cc`
  - `XerxesWrapper::accessAndRespond()` 会先把请求送进 Xerxes 拓扑。
  - `XerxesWrapper::actual_accessAndRespond()` 最终仍调用 `MemCtrl::accessAndRespond()`，说明**数据还是落在 gem5 的真实模拟内存里**，不是只算一个假延迟。
- `gem5/configs/example/xerxes_wrapper_minimal.py`
  - 已给出最小配置：`CPU/Bus -> XerxesWrapper(use_snoop=False) -> DRAMInterface`。

这意味着：

- A 区间所需的“真实存储”已经有。
- A 区间所需的“额外 CXL / 拓扑访问延迟”已经有。
- 如果只是做单区间 smoke test，现有代码几乎不用改。

## 2.2 区间 B：真实存储 + 访问延迟 + coherence 维护延迟

这一部分也不是从零开始，现有代码已经有一条可跑通的原型路径。

直接证据：

- `gem5/src/mem/xerxes_wrapper.cc`
  - `use_snoop=true` 时会实例化 Xerxes 的 `Snoop` 设备。
  - Xerxes 产生的 `INV` 包会被转发到 `CohInterface::issueInvalidate()`。
- `gem5/src/mem/coh_interface.cc`
  - `CohInterface` 可以通过 `ruby_port` 向 Ruby DMASequencer 发起 invalidate 请求。
- `gem5/configs/example/xerxes_mi_example_xerxes.py`
- `gem5/configs/example/xerxes_mesi_two_level_xerxes.py`
  - 已经给出 `XerxesWrapper -> CohInterface -> Ruby DMA -> completion` 的示例。
- `gem5/src/mem/ruby/protocol/MI_example_Xerxes-dma.sm`
- `gem5/src/mem/ruby/protocol/MESI_Two_Level_Xerxes-dma.sm`
  - 已扩展 Ruby DMA controller 的协议路径。

这说明：

- “B 区间比 A 区间多一个 coherence 维护代价”这个实验方向，现有代码是有原型基础的。
- 但这个基础目前更适合做“coherence evidence / latency injection proof-of-concept”，还不等于完整可扩展实现。

---

## 3. 为什么现有代码还不能直接满足目标方案

## 3.1 没有现成的 `x86 FS + Linux guest + 用户态 mmap 双区间` 配置

需求文档明确要：

- x86 FS 模式；
- Linux guest；
- 用户态通过 `mmap` 访问两个物理区间；
- 后续用同一个 Linux 系统下多个进程模拟多个计算节点。

但当前 Xerxes 集成里的示例基本都是：

- `SE` 模式；
- `MemTest` / `PyTrafficGen` / `RubyDirectedTester`；
- 没有现成 guest 驱动、保留内存、双区间暴露逻辑。

直接证据：

- `gem5/configs/example/xerxes_wrapper_minimal.py`
  - `Root(full_system=False, system=system)`。
- `gem5/configs/example/xerxes_mi_example_xerxes.py`
  - `Root(full_system=False, system=system)`。
- `gem5/configs/example/xerxes_mesi_two_level_xerxes.py`
  - `Root(full_system=False, system=system)`。

所以，**光靠现有 example，无法直接启动 Linux 并让 guest 用户态 mmap 两个池子**。

## 3.2 现有 B 区间示例不是“CPU 真正经由 XerxesWrapper 访问该内存池”

这是最关键的差距。

在 `xerxes_mi_example_xerxes.py` 和 `xerxes_mesi_two_level_xerxes.py` 中：

- `system.mem_ranges=[AddrRange(args.mem_size)]`
  - Ruby 所维护的系统物理内存范围是从 `0` 开始的一段范围。
- `system.mem_ctrl.dram = DDR3_1600_8x8(range=AddrRange(0x100000000, size=args.mem_size))`
  - XerxesWrapper 背后的真实存储放在 `0x100000000` 这个 alias 地址段。
- `system.coh_interface.ruby_addr_offset = 0x100000000`
  - 说明 `CohInterface` 是把 Xerxes 侧地址减去这个 offset 后，再映射回 Ruby 的真实 line 地址。
- `system.tgen.port = system.mem_ctrl.port`
  - 发起 Xerxes 流量的是 `PyTrafficGen`，不是 guest CPU。

这条路径的真实含义是：

- Ruby/cache 层持有地址 `0..mem_size` 的缓存状态；
- XerxesWrapper 侧有一个位于 `0x100000000..` 的别名物理窗口；
- Xerxes agent 访问 alias 窗口时，触发对 Ruby 中对应 line 的 back-invalidation。

所以它证明的是：

- “Xerxes 外部 agent 可以诱发 Ruby cache invalidation 并产生额外代价”。

但它**还没有证明**：

- “Linux guest 进程直接 load/store 某个物理区间时，请求路径天然就是 CPU/cache/Ruby -> XerxesWrapper -> real memory”。

而你的目标系统需要的恰恰是后者。

## 3.3 Ruby memory path 目前没有真正被 XerxesWrapper 接管

当前 Ruby 体系真正创建 memory controller 的地方是：

- `third_party/gem5/configs/ruby/Ruby.py`
  - `setup_memory_controllers()`

这个函数会根据 `system.mem_ranges` 自动创建普通 `MemCtrl` 或其他标准 memory interface，而不是 `XerxesWrapper`。

这意味着：

- 即使 CPU 通过 Ruby 访问 guest 物理内存，默认也还是走 Ruby 自己创建的 memory controllers。
- 现有 `XerxesWrapper` 示例只是旁路的、单独挂接的设备，不是整个 Ruby-backed 物理内存系统的统一后端。

因此，**现有代码没有把“guest 真实物理内存访问”全面替换成“经 Xerxes 的远端内存访问”**。

## 3.4 coherence 路径明确还是 phase-1，能力不足以支撑目标压力场景

`CohInterface` 和 `XerxesWrapper` 里已经直接写明了当前限制：

- `gem5/src/mem/coh_interface.cc`
  - `fatal_if(maxOutstandingPerLine != 1, ...)`
  - `fatal_if(pendingPkt || outstanding, ...)`
  - 这代表当前实际上只支持**单 outstanding invalidation**。
- `gem5/src/mem/xerxes_wrapper.cc`
  - `fatal_if(pkt.burst != 1, ...)`
  - 当前 invalidation 只支持**单 cache line**。

所以当前 B 区间路径不适合直接承载：

- 多核并发；
- 多进程同时打热同一远端池；
- 多 line burst invalidation；
- 高吞吐、乱序完成、重试和拥塞场景。

它更像：

- “证明机制存在”；
- “可以给 coherence 维护注入额外延迟”；
- “可做小规模实验”。

但还不是你后续评估 CXLKV HWCC 时想要的稳健后端。

## 3.5 guest 侧“保留物理区间并 mmap 给用户态”也没有现成落地

当前仓库里没有现成的：

- guest kernel driver；
- device tree / e820 / ACPI 风格的保留区间暴露逻辑；
- `/dev/cxl_pool_a`、`/dev/cxl_pool_b` 之类的 mmap 设备；
- 自动在 Linux guest 里把两段地址空间暴露给用户态测试程序的脚本。

`gem5/configs/common/FileSystemConfig.py` 里主要是伪文件系统信息生成，不是这个需求的实现点。

也就是说，**“能启动 FS”并不等于“guest 用户态就已经能拿到两块远端池地址并安全 mmap”**。

---

## 4. 对“不改现有代码能否实现”的最终判断

## 4.1 严格按需求文档理解：不能

如果目标是下面这个完整系统：

```text
x86 FS
Linux guest
两个可 mmap 的物理区间 A/B
A = 真实存储 + Xerxes 访问延迟
B = 真实存储 + Xerxes 访问延迟 + coherence 维护延迟
多个 guest 进程访问同一远端内存池
```

那么**现有代码不能直接实现**。

## 4.2 如果放宽成“先做原型验证”，则部分可以不改

可以直接利用现有代码做到的，是下面这些中间目标：

1. `SE` 下验证 A 区间的真实存储 + Xerxes 访问延迟。
2. `SE` 下验证 B 区间的 back-invalidation / coherence 维护额外代价。
3. 证明 Xerxes 与 gem5 Ruby 之间存在一条可工作的 invalidation 通路。

所以更准确地说：

- **building block 够用**
- **目标系统还差关键工程**

---

## 5. 如果要补齐完整方案，需要改哪些部分

下面按“必须改 / 高概率要改 / 可能要新建”的粒度列出。

## 5.1 第一类：FS 启动与双区间地址映射层

这一层是必须补的，因为当前没有目标系统入口。

### 必须新增

- `CXL_BETTER_SIM/Xerxes/gem5/configs/example/xerxes_x86_fs_dual_pool.py`
  - 新建一个面向 x86 FS 的主配置脚本。
  - 负责：
    - 启动 Linux guest；
    - 定义两段物理地址范围 A/B；
    - 把 A/B 分别接到两个不同参数的 Xerxes-backed memory path；
    - 配置普通内存、磁盘映像、kernel cmdline、workload。

### 高概率需要改

- `CXL_BETTER_SIM/Xerxes/gem5/configs/common/FileSystemConfig.py`
  - 如果希望 guest 看到更合理的 NUMA / node / meminfo 信息，可能需要补充。
  - 不是 A/B 实现的核心点，但做 FS 实验时通常会希望 `/sys/devices/system/node/`、`/proc/iomem` 等呈现更接近目标机器的视图。

### 可能还要新增

- guest 启动脚本、镜像构建脚本、内核命令行模板
  - 仓库里目前没有现成固定入口。
  - 可以放在 `CXL_BETTER_SIM/Xerxes/scripts/` 或 `xz_scripts/`。

## 5.2 第二类：让 guest 对 A/B 的真实内存访问真正经过 Xerxes

这是核心改造点。

当前最大问题不是“没有 XerxesWrapper”，而是“它还没有被放到 guest 的真实物理内存路径上”。

### 需要重点改的模块

- `CXL_BETTER_SIM/Xerxes/gem5/src/mem/XerxesWrapper.py`
- `CXL_BETTER_SIM/Xerxes/gem5/src/mem/xerxes_wrapper.hh`
- `CXL_BETTER_SIM/Xerxes/gem5/src/mem/xerxes_wrapper.cc`

### 为什么要改

当前 `XerxesWrapper` 的定位更像：

- 一个可直接挂在 bus 前的 memory controller；
- 或一个用于 runtime invalidation evidence 的外部 agent 后端。

但对于完整方案，需要它具备下面两种能力之一：

1. **经典内存系统路径**
   - CPU/cache/bus 访问 A/B 时，请求直接命中对应的 `XerxesWrapper`。
   - 这对 A 区间最自然。

2. **Ruby 体系内存后端路径**
   - Ruby directory / memory side 的真实物理内存访问，最终也要经过 `XerxesWrapper`。
   - 这对 B 区间尤其重要，因为当前 coherence back-end 是 Ruby。

如果保持当前设计不动，Ruby 侧真实内存访问仍然由 `Ruby.py -> setup_memory_controllers()` 创建的普通 `MemCtrl` 接管，XerxesWrapper 只能作为“旁路外部 agent”，这不符合目标系统。

### 高概率要改的文件

- `CXL_BETTER_SIM/Xerxes/third_party/gem5/configs/ruby/Ruby.py`
  - 重点函数：`setup_memory_controllers()`
  - 需要支持：
    - 对指定 `AddrRange` 使用 `XerxesWrapper` 而不是 plain `MemCtrl`；
    - 或者新增一个“Ruby memory backend uses XerxesWrapper”的分支。

### 可能要改的协议配置文件

- `CXL_BETTER_SIM/Xerxes/gem5/configs/ruby/MI_example_Xerxes.py`
- `CXL_BETTER_SIM/Xerxes/gem5/configs/ruby/MESI_Two_Level_Xerxes.py`

原因：

- 如果 B 区间最终继续依赖 Ruby DMA invalidation，那么 Ruby protocol glue 需要知道：
  - 哪些 ranges 属于 B；
  - `dma_ports` 如何与对应的 Xerxes-backed region 对应；
  - full-system 下的 CPU / IO / DMA 端口如何共同连通。

## 5.3 第三类：把 B 区间的 coherence 路径从“原型”补到“可用”

这是完整方案里技术风险最高的一段。

### 必须修改

- `CXL_BETTER_SIM/Xerxes/gem5/src/mem/CohInterface.py`
- `CXL_BETTER_SIM/Xerxes/gem5/src/mem/coh_interface.hh`
- `CXL_BETTER_SIM/Xerxes/gem5/src/mem/coh_interface.cc`

### 当前明确不足

1. 只支持单 outstanding。
2. 只支持单 line invalidate。
3. 没有成熟的队列化、乱序完成、失败恢复和拥塞处理。
4. `front_end_only=True` 仍是默认值，说明它本来就被设计成渐进式接线。

### 建议补的能力

1. 每 line / 全局 outstanding 队列。
2. 多 line invalidation 或 burst invalidation。
3. 完整 completion bookkeeping。
4. retry/backpressure 处理。
5. 更清晰的统计项：
   - Ruby 排队时间；
   - invalidation 完成时间；
   - 因脏数据回写导致的额外代价。

### 高概率还要改

- `CXL_BETTER_SIM/Xerxes/gem5/src/mem/xerxes_wrapper.cc`
  - 当前 `INV` 处理强依赖 `pkt.burst == 1`。
  - 如果 B 区间以后想更像 HWCC，就需要支持更真实的 invalidation 批量行为。

### 可能要改 Ruby 协议

- `CXL_BETTER_SIM/Xerxes/gem5/src/mem/ruby/protocol/MI_example_Xerxes-dma.sm`
- `CXL_BETTER_SIM/Xerxes/gem5/src/mem/ruby/protocol/MESI_Two_Level_Xerxes-dma.sm`
- 以及相关 `*-dir.sm` / `*-msg.sm`

原因：

- 目前 DMA 路径能证明 invalidation 能发出去并完成；
- 但如果要支持更复杂的并发或更精确的代价建模，可能要扩展消息类型、状态机和统计点。

## 5.4 第四类：guest 中把 A/B 物理区间暴露给用户态 `mmap`

这一层当前仓库基本没有现成实现，需要新增 guest 侧配套。

### 可能的实现路线

1. **保留物理地址区间 + `/dev/mem` 直接 mmap**
   - 适合最小原型；
   - 工程量小；
   - 安全性和可维护性一般。

2. **写一个最小 guest driver**
   - 比如导出 `/dev/cxl_pool_a` 和 `/dev/cxl_pool_b`；
   - `mmap` 时映射到预留物理区间；
   - 这是更干净、更稳定的路线。

### 这部分会涉及的代码位置

当前 Xerxes 仓库里没有现成文件，只能新增：

- guest kernel module 源码
  - 建议新建到 `CXL_BETTER_SIM/Xerxes/guest/` 或 `xz_scripts/guest/`
- guest 启动脚本
  - 负责 insmod、创建设备、设置权限、启动 CXLKV 进程

### simulator 侧可能配套要改

- x86 FS 配置脚本中的 kernel cmdline
  - 比如通过 `memmap=` 保留 A/B 区间，避免被 buddy allocator 当作普通内存使用。

## 5.5 第五类：双区间实验编排与统计输出

如果只是能跑起来还不够，后续评估 CXLKV 时还需要能清晰地区分 A/B 的统计。

### 建议改动

- `CXL_BETTER_SIM/Xerxes/gem5/src/mem/xerxes_wrapper.cc`
- `CXL_BETTER_SIM/Xerxes/gem5/src/mem/coh_interface.cc`
- 新的 FS 配置脚本

### 需要补的内容

1. A/B 区间分别输出统计。
2. 将 `host_inv`、bus 排队、dram 服务时间分别按区间分类。
3. 最好给 guest 可见的地址范围和 simulator 统计范围建立稳定映射，便于把 CXLKV 的 SWCC/HWCC 行为和 simulator 日志对应起来。

## 6. 对各个需求点的逐条判断

| 需求 | 现有代码是否直接满足 | 说明 |
| --- | --- | --- |
| 区间 A：真实存储 + Xerxes 访问延迟 | 基本可以 | `xerxes_wrapper_minimal.py` 已具备核心路径，但还不是 FS 双区间系统 |
| 区间 B：真实存储 + 访问延迟 + coherence 维护延迟 | 只能算原型级可以 | 现有 `CohInterface + Ruby DMA` 路径能跑，但能力有限 |
| x86 FS 启动 Linux guest | 不能直接满足 | 当前 example 都是 `full_system=False` |
| guest 用户态 `mmap` 两个区间 | 不能直接满足 | 没有预留区间和 guest 暴露机制 |
| 同一 Linux 内多进程访问共享池 | 不能直接满足 | 需要先有可稳定 mmap 的双池系统 |
| 把 A/B 映射到 CXLKV 的 SWCC/HWCC 语义 | 概念上匹配，工程上未闭环 | 需要先把 guest 可见地址和 simulator 后端路径打通 |

---

## 7. 推荐实施顺序

为了控制风险，建议按下面顺序推进，而不是一口气直接冲完整 FS + CXLKV。

### 第一步：补一个双区间 SE / synthetic 配置

目标：

- 同时有 A 区间和 B 区间；
- A 走 `use_snoop=False`；
- B 走 `use_snoop=True + CohInterface`；
- 明确输出两类请求的统计。

主要改动：

- 新增配置脚本；
- 视需要小改 `XerxesWrapper`、`CohInterface` 参数化。

### 第二步：做 x86 FS 双区间启动

目标：

- Linux guest 能启动；
- A/B 两个物理窗口真实存在；
- 先不用 CXLKV，写个最小 guest test 程序验证 load/store 和 mmap。

主要改动：

- 新增 x86 FS 配置脚本；
- 处理内存保留和 guest 侧映射。

### 第三步：把 B 区间 coherence path 补强

目标：

- 不再受限于单 outstanding / 单 line；
- 能承受后续 CXLKV 的并发访问模式。

主要改动：

- `coh_interface.*`
- `xerxes_wrapper.cc`
- 可能还有 Ruby protocol 文件。

### 第四步：接入 CXLKV

目标：

- CXLKV 的 SWCC 落到 A；
- Delta/HWCC 落到 B；
- 在 guest 里直接跑真实 workload。

在这一步之前，**不建议修改 CXLKV 的核心树逻辑**。如果 simulator 这边能把 A/B 池子以稳定 mmap 接口提供出来，CXLKV 主体大概率可以保持不动，只需要在启动配置或分配入口上绑定到不同池。

---

## 8. 最后的建议

最合理的判断不是“完全可行”或“完全不可行”，而是：

- **目标方向是可行的**；
- **现有 Xerxes 代码已经具备关键 building blocks**；
- **但离需求文档中的完整实验平台还有明显工程差距**。

最关键的三个缺口是：

1. 没有现成的 `x86 FS + Linux guest + 双区间 mmap` 入口。
2. 现有 B 区间示例是 alias + external agent invalidation 证据链，不是 guest 真实数据路径。
3. `CohInterface` 仍然只是 phase-1，实现能力不足以直接支撑高并发 HWCC 实验。

因此，后续改造时建议把注意力优先放在以下文件：

- `CXL_BETTER_SIM/Xerxes/gem5/src/mem/xerxes_wrapper.cc`
